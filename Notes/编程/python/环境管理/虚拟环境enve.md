### 1. 核心逻辑：环境隔离的必要性

Python 的第三方库默认安装在全局目录下。想象一下，如果你有两个项目：

- **项目 A** 需要 `Django 2.2`。
    
- **项目 B** 需要 `Django 4.0`。
    

由于全局环境同一时间只能存在一个版本的库，安装项目 B 的依赖就会直接覆盖掉项目 A 的依赖，导致项目 A 崩溃。虚拟环境的本质，就是在你的电脑中**开辟一个独立的文件夹**，并将 Python 解释器和 pip 路径映射到这个文件夹内。这样，每个项目都有了自己“私人的”图书馆。

---

### 2. 使用标准库 `venv` 创建环境

现在，我们通过 Python 自带的 `venv` 模块来实现这一点。

**第一步：创建环境** 打开终端，进入你的项目目录，输入以下命令：

Bash

```
# 'python -m venv' 是指令，后一个 'myenv' 是你给环境起的文件夹名字
python -m venv myenv
```

**底层逻辑拆解：** 当你运行这行代码时，Python 做了以下几件事：

1. 创建了一个名为 `myenv` 的文件夹。
    
2. 在该文件夹的 `bin`（Linux/Mac）或 `Scripts`（Windows）目录下存放了 **Python 解释器的副本或符号链接**。
    
3. 初始化了一个私有的 `site-packages` 文件夹，用于存放后续安装的库。
    

---

### 3. 激活环境：改变路径指向

创建了环境并不代表你正在使用它。你需要“激活”它，告诉系统：“从现在起，请使用这个特定文件夹里的 Python，而不是全局那个。”

- **Windows:**


```
myenv\Scripts\activate
```
    
- **macOS / Linux:**
```
source myenv/bin/activate
```
    

**关键变化：** 一旦激活，你会发现终端提示符的前面多了一个 `(myenv)` 括号。这代表你的 `PATH` 环境变量被临时修改了，`python` 指令现在指向的是 `myenv` 内部的执行文件。

### 4. 环境内的包管理与“蓝图”思想

激活环境后，你就可以像往常一样使用 `pip` 安装第三方库了。

**安装依赖：**

Bash

```
# 此时由于 PATH 已经被修改，调用的 pip 是虚拟环境内的 pip
# requests 库会被下载，并严格放置在 myenv/lib/pythonX.X/site-packages 目录中，绝不污染全局
pip install requests
```

**底层逻辑：环境的隔离与复用** 在团队协作或项目部署时，这里有一个极其核心的工程规范：**永远不要把 `myenv` 这个文件夹打包发给别人，也不要上传到 Git 等代码托管平台中。**

为什么？因为虚拟环境文件夹不仅体积庞大，而且包含了特定操作系统（Windows/Mac/Linux）下编译的二进制文件。你在 Windows 下创建的虚拟环境，直接拷给 Mac 用户是无法运行的。

那么，别人如果想运行你的项目，怎么保证库的版本一致呢？这就引入了“**蓝图（Manifest）**”机制。我们不传递环境本身，而是传递一份“说明书”，让其他开发者在自己的电脑上按照说明书**重建**环境。

**生成“说明书”：**

Bash

```
# 'pip freeze' 会扫描当前虚拟环境，列出所有已安装的库及其精确版本号
# '>' 是终端重定向符号，将扫描结果写入名为 requirements.txt 的纯文本文件中
pip freeze > requirements.txt
```

**根据“说明书”重建环境（当别人拿到你的项目时）：**

Bash

```
# 其他人创建并激活自己的虚拟环境后，执行此命令
# pip 会逐行读取 requirements.txt，批量下载并建立一模一样的环境
pip install -r requirements.txt
```

---

### 5. 退出环境：系统状态复位

当你完成了当前项目的工作，想要切换到其他项目，或者恢复全局的 Python 状态时，你需要“退出”当前的虚拟环境。

**退出命令：**

Bash

```
# 这个指令会撤销激活时对环境变量 PATH 的临时修改，恢复系统原状
deactivate
```

执行完毕后，命令行前缀的 `(myenv)` 会随之消失。当你再次输入 `python` 或 `pip`，系统又会重新指向全局环境。
